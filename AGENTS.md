## Git & PR conventions

- Branch from `main`; never commit directly to `main`.
- Branch names: `<type>/<short-desc>` (e.g. `feat/user-auth`, `fix/null-token`).
- Commit messages: imperative mood, present tense ("Add X", not "Added X").
- One logical change per commit. Run tests + formatter before committing.
- Open a PR with a summary of *what* changed and *why*; link the issue.

## Working agreement (how to behave)

These mirror the principles in [docs/best-practices.md](docs/best-practices.md):

1. **Think before coding.** State assumptions. If the request is ambiguous, ask — don't guess.
2. **Simplicity first.** The minimum code that solves the problem. No speculative abstractions.
3. **Surgical changes.** Touch only what the task requires. Don't refactor adjacent code or
   "improve" formatting you weren't asked to change.
4. **Verify, don't assert.** Run the command, read the output, *then* report. "Tests pass" requires
   you to have actually run them.
5. **Plan multi-step work.** For anything non-trivial, write the plan first (use `/plan`), then
   execute against it.

## Security & safety

- Never commit secrets. Secrets come from environment variables / `.env` (git-ignored).
- Never log credentials, tokens, or PII.
- Don't add dependencies for trivial functionality you can write in a few lines.
- Treat anything under `<sensitive paths>` as read-only unless explicitly told otherwise.

## Don't

- Don't disable tests, linters, or type checks to make something pass — fix the cause.
- Don't introduce a new framework/library without flagging the tradeoff first.
- Don't leave commented-out code or `TODO`s without an owner/issue.
